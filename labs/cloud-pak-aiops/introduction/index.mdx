Telecommunications operations manage increasingly complex environments, with multiple
network domains, interdependent services, and large volumes of events, alarms, metrics,
and topology data. In this context, a single fault may generate multiple symptoms across
different layers, making it more difficult to quickly determine both the root cause and the
actual service impact.

The challenge is often not the availability of data, but the ability to interpret that data
consistently and in a timely manner in order to reduce noise, accelerate analysis, and
improve incident prioritization. This is particularly relevant in NOC and Service Assurance
environments, where timely understanding and response are closely tied to service
continuity and quality.

This lab uses a transport optical network scenario as a reference, based on a fault occurring
in a network segment and its propagation across associated resources and services. Using
this case, the lab examines how events, metrics, and topology can be interpreted with
greater operational context to better understand the source of the issue, its scope, and its
operational impact.

This is the current lab list:

- [**Topology**](../topology-lab/1-introduction/index.mdx)
  - Create topology data
  - Use rules to merge topology elements
  - Use templates to gather resources into logical groups
  - Define applications and configure business criticality

- [**Alert Correlation**](../alert-lab/1-introduction/index.mdx)
  - Load sample event and alert data using a generic webhook and a Kafka topic
  - Implement scope-based alert correlation
  - Implement temporal alert correlation
  - Implement topological alert correlation


- [**Runbook Automation**](../runbook-automation/1-introduction/index.mdx)
  - Create an SSH integration
  - Create runbook actions
  - Create a runbook
  - Create a runbook policy
